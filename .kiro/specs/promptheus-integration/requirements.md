# Requirements Document: PROMPTHEUS Agent-Based Security Scanning

## Introduction

This document specifies the requirements for enhancing PROMPTHEUS with agent-based security scanning capabilities. The enhancement transforms PROMPTHEUS from a payload-based red-team tool into a comprehensive multi-phase security analysis platform while maintaining backward compatibility with existing functionality.

The system SHALL support three scanning modes: legacy payload-based scanning, agent-based multi-phase scanning, and hybrid mode combining both approaches. The integration SHALL preserve the existing hexagonal architecture and adapter system while adding agent orchestration, DAST validation, real-time progress tracking, and skill-based testing capabilities.

## Glossary

- **Scanner**: The unified security scanning system that orchestrates both legacy and agent-based scanning modes
- **Agent**: A specialized Claude-powered sub-agent that performs a specific security analysis phase
- **Orchestrator**: The component that manages agent lifecycle and execution order
- **Artifact**: A structured output file generated by agents (e.g., SECURITY.md, THREAT_MODEL.json)
- **DAST**: Dynamic Application Security Testing - HTTP-based validation of vulnerabilities
- **Skill**: A reusable security testing capability that can be loaded by agents
- **Progress_Tracker**: Component that provides real-time feedback during long-running scans
- **Hook**: A callback function that intercepts tool execution for security enforcement or tracking
- **Phase**: A distinct stage in the agent-based scanning workflow
- **Repository_Root**: The base directory of the codebase being scanned
- **Artifacts_Directory**: The `.promptheus/` directory where artifacts are stored
- **Severity_Threshold**: The minimum severity level for reporting issues (CRITICAL, HIGH, MEDIUM, LOW, INFO)
- **Validation_Status**: The DAST validation result (VALIDATED, FALSE_POSITIVE, PARTIAL, UNVALIDATED)
- **PR_Review**: Pull request security analysis that focuses on changed code
- **Diff_Context**: Parsed representation of code changes in a pull request
- **Chain_Analysis**: Detection of exploit chains across multiple vulnerabilities
- **Triage**: Budget-aware filtering of findings to focus on high-impact issues

## Requirements

### Requirement 1: Unified Scanner Interface

**User Story:** As a security engineer, I want a single scanner interface that supports multiple scanning modes, so that I can choose the appropriate analysis approach for my needs.

#### Acceptance Criteria

1. THE Scanner SHALL support three scanning modes: legacy, agent, and hybrid
2. WHEN a scan is initiated, THE Scanner SHALL validate the configuration before execution
3. WHEN legacy mode is selected, THE Scanner SHALL execute payload-based scanning using the existing RedTeamEngine
4. WHEN agent mode is selected, THE Scanner SHALL execute multi-phase agent-based scanning
5. WHEN hybrid mode is selected, THE Scanner SHALL execute both legacy and agent scans and merge the results
6. THE Scanner SHALL track total API costs across all phases
7. THE Scanner SHALL provide a consistent ScanResult interface regardless of mode

### Requirement 2: Agent-Based Scanning Workflow

**User Story:** As a security engineer, I want agent-based scanning to follow a structured multi-phase workflow, so that I can perform comprehensive security analysis with clear phase boundaries.

#### Acceptance Criteria

1. WHEN agent mode is enabled, THE Scanner SHALL execute phases in the following order: assessment, threat-modeling, code-review, report-generator
2. WHEN DAST is enabled, THE Scanner SHALL execute the dast phase after report-generator
3. WHEN a phase completes successfully, THE Scanner SHALL write the phase artifact to the Artifacts_Directory before starting the next phase
4. WHEN a phase fails, THE Scanner SHALL stop execution and return an error with phase context
5. THE Scanner SHALL support resuming from a specific phase using the resume_from parameter
6. THE Scanner SHALL support running a single phase using the single_subagent parameter
7. THE Scanner SHALL support skipping specific phases using the skip_subagents parameter

### Requirement 3: Agent Definitions and Configuration

**User Story:** As a developer, I want agent definitions to be configurable with appropriate tools and models, so that each phase can be optimized for its specific task.

#### Acceptance Criteria

1. THE System SHALL provide six agent definitions: assessment, threat-modeling, code-review, pr-code-review, report-generator, and dast
2. WHEN creating agent definitions, THE System SHALL load prompts from centralized prompt files
3. WHEN a CLI model override is provided, THE System SHALL apply it to all agents unless overridden by per-agent environment variables
4. WHEN DAST target URL is provided, THE System SHALL inject it into the dast agent prompt
5. WHEN threat modeling context is provided, THE System SHALL inject it into the threat-modeling agent prompt
6. THE assessment agent SHALL have access to Read, Grep, Glob, LS, and Write tools
7. THE threat-modeling agent SHALL have access to Read, Grep, Glob, Write, and Skill tools
8. THE code-review agent SHALL have access to Read, Grep, Glob, and Write tools
9. THE pr-code-review agent SHALL have access to Read, Grep, Glob, and Write tools
10. THE report-generator agent SHALL have access to Read and Write tools
11. THE dast agent SHALL have access to Read, Write, Skill, and Bash tools

### Requirement 4: Real-Time Progress Tracking

**User Story:** As a security engineer, I want real-time progress updates during long-running scans, so that I can monitor scan activity and understand what the system is doing.

#### Acceptance Criteria

1. WHEN a scan is running, THE Progress_Tracker SHALL display the current phase name and number
2. WHEN a tool is executed, THE Progress_Tracker SHALL log the tool name and relevant parameters
3. WHEN a file is read, THE Progress_Tracker SHALL track the file path in the files_read set
4. WHEN a file is written, THE Progress_Tracker SHALL track the file path in the files_written set
5. WHEN a phase completes, THE Progress_Tracker SHALL display phase duration, tool count, and file operation counts
6. WHEN debug mode is enabled, THE Progress_Tracker SHALL display assistant narration text
7. WHEN a sub-agent starts, THE Progress_Tracker SHALL announce the phase transition
8. WHEN a sub-agent stops, THE Progress_Tracker SHALL display completion summary with created artifacts

### Requirement 5: Security Hooks and Guardrails

**User Story:** As a security engineer, I want the scanner to enforce security boundaries and restrictions, so that agents cannot perform unsafe operations or access restricted resources.

#### Acceptance Criteria

1. WHEN the dast phase is active, THE System SHALL block database manipulation tools (sqlite3, psql, mysql, etc.)
2. WHEN any agent attempts to read from infrastructure directories, THE System SHALL skip the operation and return a skip message
3. WHEN any agent attempts to grep infrastructure directories, THE System SHALL inject exclude patterns to filter them out
4. WHEN the dast phase attempts to write files, THE System SHALL only allow writes to DAST_VALIDATION.json or /tmp/*
5. WHEN the pr-code-review phase attempts to write files, THE System SHALL only allow writes to PR_VULNERABILITIES.json
6. WHEN any agent attempts to access files outside Repository_Root, THE System SHALL block the operation and return an error
7. WHEN the pr-code-review phase attempts pathless grep, THE System SHALL inject a safe default scope derived from changed files
8. THE System SHALL maintain a list of excluded directories per language (node_modules, venv, .git, etc.)
9. THE System SHALL provide different exclusion lists for different phases (DAST needs access to .claude/skills/)

### Requirement 6: Artifact Management

**User Story:** As a security engineer, I want scan artifacts to be stored in a consistent location with proper validation, so that I can review and process scan results reliably.

#### Acceptance Criteria

1. THE System SHALL store all artifacts in the Artifacts_Directory (.promptheus/)
2. THE assessment phase SHALL create SECURITY.md artifact
3. THE threat-modeling phase SHALL create THREAT_MODEL.json artifact
4. THE code-review phase SHALL create VULNERABILITIES.json artifact
5. THE pr-code-review phase SHALL create PR_VULNERABILITIES.json artifact
6. THE report-generator phase SHALL create scan_results.json artifact
7. THE dast phase SHALL create DAST_VALIDATION.json artifact
8. WHEN an artifact is written, THE System SHALL validate JSON structure for JSON artifacts
9. WHEN JSON validation fails, THE System SHALL attempt to auto-fix common issues (markdown wrapping, trailing commas)
10. WHEN PR review completes, THE System SHALL update THREAT_MODEL.json and VULNERABILITIES.json with new findings

### Requirement 7: DAST Validation

**User Story:** As a security engineer, I want to validate static analysis findings with dynamic testing, so that I can confirm exploitability and reduce false positives.

#### Acceptance Criteria

1. WHEN DAST is enabled, THE Scanner SHALL configure the dast agent with target URL and timeout
2. WHEN DAST is enabled, THE Scanner SHALL sync DAST skills to the Artifacts_Directory before the dast phase
3. WHEN DAST is enabled and accounts file is provided, THE Scanner SHALL copy it to DAST_TEST_ACCOUNTS.json
4. WHEN the dast agent validates a vulnerability, THE System SHALL match the vulnerability CWE to available skills
5. WHEN no matching skill exists for a CWE, THE System SHALL mark the vulnerability as UNVALIDATED
6. WHEN a skill successfully exploits a vulnerability, THE System SHALL mark it as VALIDATED
7. WHEN a skill proves security controls are working, THE System SHALL mark the vulnerability as FALSE_POSITIVE
8. WHEN a skill produces mixed results, THE System SHALL mark the vulnerability as PARTIAL
9. THE dast agent SHALL only interact with the target via HTTP requests
10. THE dast agent SHALL NOT have direct database access

### Requirement 8: Skill System Integration

**User Story:** As a security engineer, I want agents to load and use security testing skills, so that they can perform specialized testing for specific vulnerability types.

#### Acceptance Criteria

1. THE System SHALL discover skills in the promptheus/skills/ directory
2. THE System SHALL support two skill categories: dast and threat-modeling
3. WHEN threat-modeling phase starts, THE System SHALL sync threat-modeling skills to .claude/skills/threat-modeling/
4. WHEN dast phase starts, THE System SHALL sync dast skills to .claude/skills/dast/
5. WHEN skills are synced, THE System SHALL preserve the directory structure and SKILL.md files
6. THE System SHALL validate skill structure before deployment
7. THE System SHALL skip invalid skills with a warning message

### Requirement 9: PR Review Flow

**User Story:** As a security engineer, I want to perform security-focused code review on pull requests, so that I can identify vulnerabilities introduced or enabled by code changes.

#### Acceptance Criteria

1. WHEN PR review is initiated, THE Scanner SHALL parse the diff and extract changed files and hunks
2. WHEN PR review is initiated, THE Scanner SHALL build focused diff context by prioritizing security-relevant files
3. WHEN PR review is initiated, THE Scanner SHALL extract relevant threats and vulnerabilities from baseline artifacts
4. WHEN PR review is initiated, THE Scanner SHALL generate exploit hypotheses based on diff and baseline context
5. WHEN PR review runs multiple attempts, THE Scanner SHALL carry forward candidate findings between attempts
6. WHEN PR review attempts disagree, THE Scanner SHALL run a verifier pass to adjudicate contradictions
7. WHEN PR review completes, THE Scanner SHALL merge findings from all attempts using chain analysis
8. WHEN PR review completes, THE Scanner SHALL deduplicate findings based on chain IDs
9. WHEN PR review completes, THE Scanner SHALL filter out findings that exist in baseline
10. THE pr-code-review agent SHALL only analyze changed files and their security-relevant context

### Requirement 10: Chain Analysis and Consensus

**User Story:** As a security engineer, I want the system to detect exploit chains and build consensus across multiple review attempts, so that I can identify high-impact multi-step vulnerabilities.

#### Acceptance Criteria

1. WHEN analyzing findings, THE System SHALL extract chain IDs at three levels: exact, family, and flow
2. WHEN multiple attempts produce findings, THE System SHALL track support counts for each chain ID
3. WHEN merging findings, THE System SHALL prefer findings with higher support counts
4. WHEN a finding appears in multiple attempts, THE System SHALL consolidate it into a single canonical finding
5. WHEN chain analysis detects command builder signals in diff, THE System SHALL focus on option injection chains
6. WHEN chain analysis detects path parser signals in diff, THE System SHALL focus on path traversal chains
7. WHEN chain analysis detects auth/privilege signals in diff, THE System SHALL focus on privilege escalation chains
8. THE System SHALL use family-level chain IDs as the default consensus mode

### Requirement 11: Triage and Budget Management

**User Story:** As a security engineer, I want the system to optimize resource usage by triaging findings, so that I can stay within budget constraints while focusing on high-impact issues.

#### Acceptance Criteria

1. WHEN triage is enabled, THE System SHALL estimate token costs for remaining PR review attempts
2. WHEN estimated costs exceed budget, THE System SHALL reduce the number of attempts
3. WHEN triage reduces attempts, THE System SHALL prioritize high-severity findings
4. WHEN triage is active, THE System SHALL log budget decisions for observability
5. THE System SHALL support configurable budget thresholds per scan type

### Requirement 12: Data Models and Schemas

**User Story:** As a developer, I want consistent data models for security issues and scan results, so that I can reliably process and display scan outputs.

#### Acceptance Criteria

1. THE SecurityIssue model SHALL include id, severity, title, description, file_path, line_number, and code_snippet fields
2. THE SecurityIssue model SHALL support optional DAST validation fields: validation_status, dast_evidence, exploitability_score, validated_at
3. THE SecurityIssue model SHALL support optional PR review fields: finding_type, attack_scenario, evidence
4. THE ScanResult model SHALL include repository_path, issues, files_scanned, scan_time_seconds, and total_cost_usd fields
5. THE ScanResult model SHALL support optional DAST metrics: dast_enabled, dast_validation_rate, dast_false_positive_rate, dast_scan_time_seconds
6. THE Severity enum SHALL support five levels: CRITICAL, HIGH, MEDIUM, LOW, INFO
7. THE ValidationStatus enum SHALL support four statuses: VALIDATED, FALSE_POSITIVE, UNVALIDATED, PARTIAL
8. THE ScanResult model SHALL provide methods to filter issues by severity and validation status

### Requirement 13: Error Handling and Recovery

**User Story:** As a security engineer, I want clear error messages and recovery options when scans fail, so that I can troubleshoot issues and resume work efficiently.

#### Acceptance Criteria

1. WHEN a phase fails, THE Scanner SHALL capture the error message and phase context
2. WHEN a phase fails, THE Scanner SHALL mark the phase as failed in the result
3. WHEN a phase fails, THE Scanner SHALL preserve existing artifacts for inspection
4. WHEN configuration validation fails, THE Scanner SHALL return a descriptive error before starting the scan
5. WHEN DAST target is unreachable, THE Scanner SHALL warn the user and offer to continue with SAST only
6. WHEN artifact parsing fails, THE Scanner SHALL log the raw artifact content for debugging
7. WHEN a timeout occurs, THE Scanner SHALL force-close the Claude client and report the timeout
8. THE Scanner SHALL support resuming from a failed phase using the --resume-from flag

### Requirement 14: Diff Context Extraction

**User Story:** As a security engineer, I want the system to extract and focus on security-relevant code changes, so that PR reviews are efficient and accurate.

#### Acceptance Criteria

1. WHEN parsing a diff, THE System SHALL extract changed files, hunks, and line numbers
2. WHEN building focused diff context, THE System SHALL score files based on security relevance
3. WHEN building focused diff context, THE System SHALL prioritize files with security-related path hints (auth, permission, config, etc.)
4. WHEN building focused diff context, THE System SHALL deprioritize documentation and test files
5. WHEN building focused diff context, THE System SHALL limit to a maximum number of files and hunk lines
6. WHEN focused diff would drop security-relevant files, THE System SHALL fail closed and require splitting the review
7. WHEN diff contains new files, THE System SHALL use higher line limits since they cannot be read from disk
8. THE System SHALL write individual diff files to DIFF_FILES/ subdirectory for agent access

### Requirement 15: Hypothesis Generation

**User Story:** As a security engineer, I want the system to generate exploit hypotheses from code changes, so that PR reviews can focus on high-impact attack scenarios.

#### Acceptance Criteria

1. WHEN PR review starts, THE System SHALL generate 3-8 exploit hypotheses using an LLM
2. WHEN generating hypotheses, THE System SHALL consider changed files, diff hunks, relevant threats, and baseline vulnerabilities
3. WHEN generating hypotheses, THE System SHALL focus on high-impact chains: auth bypass, command injection, path traversal, credential exfiltration
4. WHEN generating hypotheses, THE System SHALL include file/line anchor references
5. WHEN hypothesis generation times out, THE System SHALL continue with a default message and log a warning
6. THE System SHALL normalize hypothesis output into bullet format for consistency

### Requirement 16: Finding Refinement and Verification

**User Story:** As a security engineer, I want the system to refine and verify findings using LLM-based quality passes, so that I can reduce false positives and consolidate duplicate findings.

#### Acceptance Criteria

1. WHEN refining findings, THE System SHALL use an LLM-only quality pass to filter speculative findings
2. WHEN refining findings, THE System SHALL consolidate duplicate findings into canonical exploit chains
3. WHEN refining findings, THE System SHALL drop findings below the severity threshold
4. WHEN refining findings, THE System SHALL preserve concrete exploit primitives over generic hardening observations
5. WHEN running verifier mode, THE System SHALL adjudicate contradictions between attempts
6. WHEN running verifier mode, THE System SHALL require concrete source-to-sink evidence
7. WHEN refinement times out, THE System SHALL retain unrefined findings and log a warning

### Requirement 17: State Management and Observability

**User Story:** As a security engineer, I want the system to track scan state and provide observability into the scanning process, so that I can understand what happened and debug issues.

#### Acceptance Criteria

1. THE System SHALL record scan state including repository, branch, commit, and timestamp
2. THE System SHALL track attempt counts, finding counts, and chain support counts
3. THE System SHALL track blocked tool calls and out-of-repo access attempts
4. THE System SHALL log revalidation attempts and core evidence presence
5. THE System SHALL record consensus mode and weak consensus triggers
6. THE System SHALL provide merge statistics for PR review findings
7. THE System SHALL track warnings and errors throughout the scan lifecycle

### Requirement 18: Backward Compatibility

**User Story:** As an existing PROMPTHEUS user, I want the integration to preserve existing functionality, so that I can continue using legacy mode without changes.

#### Acceptance Criteria

1. WHEN no mode is specified, THE Scanner SHALL default to legacy mode
2. WHEN legacy mode is used, THE Scanner SHALL produce identical results to the pre-integration version
3. THE Scanner SHALL support all existing CLI flags and options
4. THE Scanner SHALL support all existing adapters: local, REST, and Slack
5. THE Scanner SHALL support the existing payloads.json format
6. THE Scanner SHALL maintain the existing hexagonal architecture
7. THE Scanner SHALL not require any changes to existing user workflows

### Requirement 19: Configuration and Environment

**User Story:** As a developer, I want flexible configuration options, so that I can customize scanner behavior for different environments and use cases.

#### Acceptance Criteria

1. THE System SHALL support per-agent model overrides via environment variables (PROMPTHEUS_<AGENT>_MODEL)
2. THE System SHALL support CLI model override that applies to all agents
3. THE System SHALL support configurable max turns per agent
4. THE System SHALL support configurable timeout values for PR review and DAST
5. THE System SHALL support configurable severity thresholds
6. THE System SHALL support configurable excluded directories per language
7. THE System SHALL load configuration from project settings when available

### Requirement 20: Reporting and Output

**User Story:** As a security engineer, I want scan results in multiple formats, so that I can integrate with different tools and workflows.

#### Acceptance Criteria

1. THE System SHALL support JSON output format for programmatic processing
2. THE System SHALL support Markdown output format for human readability
3. THE ScanResult SHALL include summary statistics: total issues, counts by severity
4. WHEN DAST is enabled, THE ScanResult SHALL include DAST metrics: validation rate, false positive rate, validated count
5. THE System SHALL include warnings in the output when non-fatal issues occur
6. THE System SHALL display progress and completion messages to the console during scans
7. THE System SHALL format output using Rich console for improved readability
